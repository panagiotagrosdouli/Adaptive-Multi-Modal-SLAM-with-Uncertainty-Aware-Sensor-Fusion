# Claims and Evidence Ledger

This ledger protects the repository from overstating unfinished work. Every scientific or engineering claim should map to evidence: source code, tests, experiment configs, generated result files, documentation, or citations.

## Status vocabulary

| Status | Meaning | Allowed public wording |
| --- | --- | --- |
| Implemented | Feature has source code and tests or a reproducible smoke experiment. | "Implemented baseline", "implemented utility", "tested component" |
| Prototype | Feature has partial code or scaffold but is not complete enough for benchmark claims. | "Prototype", "experimental scaffold", "under active development" |
| Planned | Feature is documented but not implemented. | "Planned", "future work", "not yet implemented" |
| Pending | Experiment target exists but no committed metric artifact is available. | "Pending evaluation", "results not yet reported" |
| Unsupported | Claim has no evidence and must be removed or rewritten. | Do not use |

## Rules for README, website, and paper text

1. Do not use "state-of-the-art" unless a verified benchmark comparison exists.
2. Do not report real dataset numbers unless generated result files and manifests are committed.
3. Do not call a scaffold "implemented" unless tests or smoke experiments exist.
4. Synthetic examples must be labeled synthetic.
5. Risk scores must be called diagnostic unless calibrated against held-out failure labels.
6. Dataset documentation must link to official dataset sources and must not redistribute restricted data.
7. Website figures must be generated by committed scripts or clearly marked as conceptual diagrams.

## Evidence required for future upgrades

### To upgrade a backend from Prototype to Implemented

- Source code for prediction/update or factor residuals.
- Unit tests for covariance propagation, gating, and residual dimensions.
- At least one deterministic synthetic experiment.
- Logged metrics and manifest.
- Documentation of assumptions.

### To upgrade a dataset from Pending to Evaluated

- Loader or adapter code.
- Timestamp association and calibration documentation.
- Reproducible config.
- Metric JSON.
- Result manifest with commit SHA and dataset sequence identifier.
- Script-generated figure or table.

### To upgrade risk score to calibrated probability

- Ground-truth failure labels or reproducible failure proxy.
- Calibration curves.
- Expected calibration error or similar metric.
- Held-out validation sequences.
- Documentation of failure definition.
